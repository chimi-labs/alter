BUG: Canvas modify_column accepts arbitrary fields via setattr without validation
SEVERITY: Medium
STATUS: Open

REPRODUCTION:
  # From the canvas UI or any HTTP client:
  POST /propose
  {
    "op": "modify_column",
    "table": "users",
    "column": "email",
    "updates": {
      "primary_key": true,
      "inherited": true,
      "extra_kwargs": {"malicious": "data"}
    }
  }
  # All three fields are set on the Column object — no validation, no whitelist.

EXPECTED:
  Only a known set of user-modifiable column properties should be accepted.
  Unknown or read-only fields should be rejected with an error.

ROOT CAUSE:
  canvas/server.py lines 565-567:
    for k, v in updates.items():
        if hasattr(col, k):
            setattr(col, k, v)

  The `hasattr()` check prevents AttributeError but does NOT validate:
  - Whether the field SHOULD be user-editable
  - Whether the value is valid for that field type
  - Whether the change is consistent (e.g. two PKs in one table)

  The canvas JS frontend only sends valid fields, so this doesn't manifest
  in normal use. But the HTTP endpoint is unauthenticated and local — any
  process on the machine could send a malformed request.

FIX PLAN:
  1. Define a whitelist of modifiable column fields:
       _MODIFIABLE_COL_FIELDS = {
           "name", "type", "nullable", "unique", "default",
           "max_length", "index", "foreign_key",
       }
  2. Filter updates against the whitelist before applying:
       for k, v in updates.items():
           if k in _MODIFIABLE_COL_FIELDS and hasattr(col, k):
               setattr(col, k, v)
  3. Optionally validate values:
     - `type` must exist in TYPE_MAP or schema.enums
     - `nullable` and `unique` must be bool
     - `max_length` must be int or None
     This can use Pydantic's own field validators if the Column model
     already has them.
  4. Handle the `name` field specially — it's a rename, which also requires
     updating relations and foreign_key references (same logic as
     mcp_server.py rename_entity).

  Same pattern should apply to `add_column` (line 547):
    col = Column(**col_data)
  Pydantic validates this automatically, so it's less risky, but unknown
  fields in col_data are silently ignored. Consider logging unrecognised
  keys.

VERIFICATION:
  - Send a POST /propose with `updates: {"inherited": true}` → should be
    ignored (not in whitelist).
  - Send `updates: {"type": "nonexistent"}` → should reject or warn.
  - Send `updates: {"nullable": false}` → should succeed.

TESTS:
  - Test that non-whitelisted fields are silently ignored.
  - Test that whitelisted fields are applied correctly.
  - (Optional) Test that invalid type values are rejected.
