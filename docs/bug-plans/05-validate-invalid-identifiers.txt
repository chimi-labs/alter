BUG: validate_schema accepts invalid SQL/Python identifiers as table/column names
SEVERITY: Medium
STATUS: Open

REPRODUCTION:
  # Via MCP or direct schema construction:
  add_table(name="select")       # SQL reserved word
  add_table(name="123users")     # starts with digit
  add_table(name="user-name")    # contains hyphen
  add_column(table="users", name="from", type="string")  # SQL keyword

  $ alter validate    # passes with no errors
  $ alter export      # generates invalid SQL:
  #   CREATE TABLE select (...)   — syntax error
  #   CREATE TABLE 123users (...) — syntax error

EXPECTED:
  `alter validate` should flag these as errors or warnings:
  - SQL reserved words → error (breaks DDL export)
  - Names starting with digits → error (invalid identifier)
  - Names with special characters → error (breaks both SQL and Python code)

ROOT CAUSE:
  validate.py only checks for:
  - Empty names (lines 78-83, 90-95)
  - Duplicate names (lines 97-102)
  - Unknown types (lines 106-113)

  There is no regex or keyword check on the name string itself. Any
  non-empty string passes validation.

FIX PLAN:
  1. Define a SQL identifier regex:
       import re
       _VALID_IDENTIFIER_RE = re.compile(r'^[a-zA-Z_][a-zA-Z0-9_]*$')

  2. Define a set of common SQL reserved words (doesn't need to be
     exhaustive — focus on the ones that actually break DDL):
       _SQL_RESERVED = {
           "select", "from", "where", "insert", "update", "delete",
           "create", "drop", "alter", "table", "column", "index",
           "primary", "foreign", "key", "references", "constraint",
           "unique", "not", "null", "default", "check", "in", "and",
           "or", "order", "group", "by", "having", "limit", "offset",
           "join", "on", "as", "set", "values", "into",
       }

  3. Add two checks after the empty-name check for both tables and columns:

     For tables (after line 83):
       if not _VALID_IDENTIFIER_RE.match(table.name):
           issues.append(ValidationIssue(
               severity="error", table=table.name,
               message=f"Table name '{table.name}' is not a valid SQL identifier "
                       f"(must start with a letter or underscore, contain only "
                       f"alphanumeric characters and underscores)",
           ))
       elif table.name.lower() in _SQL_RESERVED:
           issues.append(ValidationIssue(
               severity="warning", table=table.name,
               message=f"Table name '{table.name}' is a SQL reserved word — "
                       f"may cause issues with DDL export and some databases",
           ))

     For columns (after line 95):
       Same pattern with column=col.name.

  4. Use "error" for structurally invalid names (digits, hyphens) since
     these WILL break generated code. Use "warning" for SQL reserved words
     since some databases handle them (Postgres allows quoting) but they're
     still risky.

VERIFICATION:
  $ alter validate  # with table named "select"
  # Should show: ⚠  Warning: Table name 'select' is a SQL reserved word

  $ alter validate  # with column named "123bad"
  # Should show: ✗  Error: Column name '123bad' is not a valid SQL identifier

TESTS:
  - Test valid names pass (users, user_profile, UserRole).
  - Test SQL keywords produce warnings.
  - Test digit-starting names produce errors.
  - Test hyphenated names produce errors.
  - Test underscore-starting names pass (common for internal columns).
